Business Scenario

A mid-size European bank currently uses a legacy SAS-based scoring model to assess personal loan applications. The model has not been reviewed in three years and the risk team has no insight into why individual applications are declined. Regulators are increasingly asking for explainability documentation, and the model risk team wants to understand error patterns across customer segments before any production migration to Azure.

Your task: Rebuild this workflow end-to-end in Azure Machine Learning — from data ingestion to a live scoring endpoint — and provide the risk team with a Responsible AI dashboard to support model governance.

0Get the Dataset

Step 0.1

Choose & Download Your Dataset

Three sources are available — pick the one that suits you. All contain 1,000 rows of German credit applicants classified as good or bad risk. The sources differ in the number of features and how easy the column names are to work with.

  • Kaggle (recommended): kaggle.com/datasets/kabure/german-credit-data-with-risk — 11 readable columns including Risk target. Requires free Kaggle account. Best for RAI Dashboard readability.
  • OpenML (no login needed): openml.org/d/31 — full 20-feature version, target column named class. Download directly as CSV without an account.
  • UCI Python package (code-first): Load directly in a notebook with from ucimlrepo import fetch_ucirepo; ds = fetch_ucirepo(id=144) — 20 features, no manual download needed.
Kaggle: 11 cols + RiskOpenML: 20 cols + classUCI: fetch_ucirepo(id=144)
Step 0.2

Upload to AML & Create an MLTable Asset

Upload the CSV to your AML workspace and register it as an MLTable data asset. MLTable is required for both AutoML and the RAI Dashboard — a plain CSV path will not work for either.

  • In AML Studio: Data → Create → From local files, select type Table (mltable)
  • Alternatively, place an MLTable definition file alongside your CSV and register via SDK: ml_client.data.create_or_update(...)
  • Name it german-credit with version 1 for easy reference in later steps
  • Verify: preview should show all 1,000 rows and the expected columns including your target (Risk or class depending on source)

Step 1.1

Provision your AML Workspace

Create a Compute Instance for development and a Compute Cluster (1–2 nodes, Standard_DS3_v2) for pipeline jobs. Your german-credit MLTable data asset from Step 0 will be used as input throughout the pipeline.

2Build a Pipeline with Components

Step 2.1

Data Prep Component

Write a Python script that reads the MLTable, handles missing values, encodes categorical features (e.g. credit history, purpose), and splits data into train/test sets. Wrap it as a reusable AML Component using either a YAML spec or the @command_component decorator.

  • Outputs: train_data and test_data as URI_FOLDER
  • Log preprocessing stats with MLflow
ComponentYAMLMLflow
Step 2.2

Training Component

Train a Random Forest classifier using scikit-learn. Use MLflow autologging to capture accuracy, AUC, and feature importances. Register the model in the AML Model Registry in MLflow format (required for RAI Dashboard).

  • Use mlflow.sklearn.autolog()
  • Register with mlflow.register_model()
TrainingMLflowModel Registry
Step 2.3

Assemble the Pipeline

Connect the two components using the @dsl.pipeline decorator. Submit the pipeline job to your Compute Cluster. Verify you can see both steps in AML Studio and inspect the run metrics for each component.

@dsl.pipelinePipeline JobSDK v2

3Run AutoML for Comparison

Step 3.1

AutoML Classification Job

Configure an AutoML classification job on the same train dataset. Set primary metric to AUC_weighted (appropriate for the 70/30 imbalance), limit to 15 trials, and enable model explainability. Compare the best AutoML model against your manual Random Forest.

  • Primary metric: AUC_weighted
  • Enable enable_model_explainability=True
  • Inspect the featurization report and guardrails output
AutoMLAUC_weightedFeaturizationGuardrails

4Responsible AI Dashboard

Step 4.1

Generate the RAI Dashboard

Using AML Studio UI, generate a RAI Dashboard for your registered Random Forest model. Enable Error Analysis, Model Explanations, and Counterfactuals. Use the train and test datasets you registered as MLTable assets.

🔍 RAI Analysis Tasks — answer these questions using the dashboard
  • Which cohort of applicants has the highest model error rate? (Hint: use Error Analysis tree, filter by age or credit_amount)
  • What are the top 3 features driving the model globally? Do they match domain intuition from a risk management perspective?
  • Pick one falsely rejected applicant. Use Local Explanations to understand which features drove the incorrect decision.
  • Use Counterfactuals: what is the minimum change to duration or credit_amount that would flip a "Bad" prediction to "Good"?
  • Governance question: How would you document these RAI findings to satisfy model risk governance requirements in a regulated financial institution?

5Deploy to an Online Endpoint

Step 5.1

Create a Managed Online Endpoint

Deploy your registered Random Forest model to a managed online endpoint using SDK v2 or Studio. Start with a single blue deployment handling 100% of traffic. Test it by sending a JSON payload with a sample applicant profile.

  • Instance type: Standard_DS2_v2, instance count: 1
  • Test with ml_client.online_endpoints.invoke()

🔧 Common Blockers & How to Resolve Them
RAI Dashboard fails to generate
Check that your model is registered as MLflow format with sklearn flavor. AutoML models are not directly supported — use your manual Random Forest. Datasets must be MLTable format, not CSV.
Pipeline component input/output mismatch
Verify your Output type in the component spec matches what downstream steps expect as Input. Use URI_FOLDER for directories. Check the .outputs property when connecting components in the pipeline function.
AutoML job fails with MLTable error
AutoML requires MLTable, not URI_FILE or raw CSV. Create an MLTable definition file (MLTable) in the same folder as your CSV, then register as an MLTable data asset.
learn.microsoft.com → how-to-create-data-assets
Online endpoint times out or returns 502
Check deployment logs in Studio (Endpoints → your endpoint → Deployment logs). For MLflow models, no scoring script is needed — if you added one, it may conflict. Ensure instance count ≥ 1 and deployment is in "Succeeded" state before invoking.
Can't see MLflow metrics in Studio
You don't need to set mlflow.set_tracking_uri() in AML — it's set automatically. Ensure mlflow.start_run() is used, or use autolog(). Look in the job's Metrics tab, not the Experiment panel.
Component environment build takes too long
Use a curated AML environment as your base image (e.g. AzureML-sklearn-1.5-ubuntu22.04-py38-cpu) rather than building from scratch. This avoids a full Docker rebuild on every run.
⚠ Known Constraints to Be Aware Of
  • The RAI Dashboard currently only supports scikit-learn models registered in MLflow format. AutoML models cannot be directly used — use your pipeline-trained Random Forest.
  • RAI Dashboard test datasets must have fewer than 5,000 rows for the visualization UI. The German Credit dataset (1,000 rows) is fine.
  • Blue/green deployment traffic split requires both deployments to be under the same endpoint. You cannot split traffic between two separate endpoints.
  • Free/trial Azure subscriptions have limited compute quota — if cluster nodes aren't provisioning, check quota in the Azure portal under Subscriptions → Usage + quotas.
